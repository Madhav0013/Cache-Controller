# Cache Controller — Error Log

| # | Phase | Error | Root Cause | Resolution | Date |
|---|-------|-------|------------|------------|------|
| 1 | Smoke Test | FP3 assertion: "FSM in DONE but cpu_ready not asserted" + FP2: "cpu_ready asserted outside of DONE state" | `cpu_ready` was registered in `always_ff` DONE case, creating a 1-cycle pipeline delay. When `state == DONE`, `cpu_ready` was still 0 (set in previous state's logic). When `cpu_ready` rose, state had already moved to IDLE. | Changed `cpu_ready` from registered to combinational: `assign cpu_ready = (state == DONE);`. Removed from `always_ff` reset, default, and DONE case. | 2026-05-21 |
| 2 | Smoke Test | Write went to addr 0xC678 instead of 0x0100; read went to 0x6CFC instead of 0x0100. Scoreboard showed false-positive PASS (both addresses uninitialized → expected 0 = got 0). | `local::` scoping bug in `randomize() with {}` inline constraints. Inside the constraint block, unqualified `addr` resolves to `t.addr` (class member), not the task argument. So `t.addr == addr` becomes the tautology `t.addr == t.addr`, providing no constraint. | Added `local::` prefix: `t.addr == local::addr; t.wdata == local::data;` in both `write_addr()` and `read_addr()` tasks. | 2026-05-21 |
| 3 | Smoke Test | Hit detection used live `cpu_addr` instead of registered `req_index_reg`/`req_tag_reg` in `cache_hit` and next_state dirty check. | The combinational `cache_hit` and next_state TAG_CHECK dirty check used `req_index`/`req_tag` derived from live `cpu_addr`. If the CPU changed `cpu_addr` after the request pulse, hit/miss detection could be wrong. | Changed `cache_hit` to use `req_index_reg`/`req_tag_reg`. Changed next_state dirty check to use `req_index_reg`/`dirty_array[req_index_reg]`. | 2026-05-21 |
| 4 | Eviction Test | `READ MISMATCH: addr=0x0100 expected=0xaaaaaaaa got=0x00000000`. After dirty eviction and re-read, data returned 0 instead of the previously written value. | `mem_ready` handshake leak: In WRITEBACK's `if (mem_ready)` branch, the cache immediately issued a new read `mem_req`. Since `mem_ready` was still high from the writeback response, the ALLOCATE state consumed the stale `mem_ready` on its first cycle and read stale `mem_rdata`. The memory never had a chance to serve the new read. | Removed the premature read request from WRITEBACK's `mem_ready` branch. Now WRITEBACK drops `mem_req` when writeback completes, creating a 1-cycle handshake gap. ALLOCATE then issues its own fresh read request. Also relaxed FP4 assertion to exclude the transition cycle: `(state == WRITEBACK && !mem_ready) \|-> (mem_req && mem_wr)`. | 2026-05-21 |
